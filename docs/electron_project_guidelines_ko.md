# Electron 프로젝트 공통 가이드

이 문서는 macOS와 Windows를 함께 지원하는 Electron 프로젝트를 설계하고 구현할 때 재사용할 수 있는 공통 기준서다.

이 문서의 목적은 두 가지다.

- 팀 내부에서 Electron 프로젝트의 기본 설계 기준을 통일한다.
- AI 모델에게 참고 문서로 제공해, 보안과 구조가 무너진 초안을 만들지 않도록 한다.

이 문서는 PRD가 아니다. 기능 요구사항 문서가 아니라 Electron 프로젝트를 만들 때 공통적으로 적용할 개발 기준 문서다.

---

## 1. 이 문서를 읽는 방법

이 문서의 규칙 수준은 아래 세 단계로 구분한다.

- `필수`: 특별한 이유가 없으면 반드시 지켜야 하는 기준
- `권장`: 대부분의 프로젝트에서 따르는 것이 좋은 기준
- `선택`: 프로젝트 성격에 따라 적용 여부가 달라지는 기준

중요한 전제:

- 이 문서는 "로컬 파일 처리형 데스크톱 앱"에 가장 잘 맞는다.
- 원격 웹 콘텐츠를 핵심 UI로 쓰는 앱, 스토어 배포 앱, 멀티윈도우 앱은 별도 분기 기준을 함께 봐야 한다.
- 문서의 어떤 항목도 프로젝트 맥락을 무시한 절대 규칙으로 해석하면 안 된다.

---

## 2. 적용 대상과 비적용 대상

### 2.1 이 문서에 잘 맞는 프로젝트

- PDF, 이미지, 영상, 문서 등 로컬 파일을 열고 가공하는 앱
- Electron `main / preload / renderer` 구조를 쓰는 앱
- macOS와 Windows에서 공통 UX를 제공해야 하는 앱
- 파일 열기, 저장, 내보내기, 작업 이력, 설정 저장이 중요한 앱

### 2.2 그대로 적용하면 안 되는 프로젝트

- 원격 웹사이트를 거의 그대로 감싸는 shell형 앱
- 외부 웹 콘텐츠를 다수 임베드하는 앱
- Mac App Store 또는 Microsoft Store 전용 배포를 목표로 하는 앱
- 창이 여러 개이고 각 창의 역할이 크게 다른 복합 워크스테이션 앱

이 경우에는 이 문서를 기본 베이스로 쓰되 아래를 별도로 보강해야 한다.

- 원격 콘텐츠 보안 모델
- 스토어 심사 및 샌드박스 제약
- 멀티윈도우 상태 동기화
- 배포 채널별 업데이트 전략

---

## 3. 프로젝트 기본 원칙

### 3.1 프로세스 역할 분리

- `필수`: `main`은 앱 생명주기, 창 생성, 파일 시스템, 네이티브 API, 메뉴, 업데이트를 담당한다.
- `필수`: `preload`는 렌더러에 필요한 최소 API만 안전하게 노출한다.
- `필수`: `renderer`는 UI와 사용자 상호작용에 집중한다.
- `권장`: 무거운 작업은 UI 흐름과 분리된 작업 단위로 관리한다.

### 3.2 권한 최소화

- `필수`: 렌더러에 Node.js 권한을 직접 광범위하게 열지 않는다.
- `필수`: 파일 열기/저장/폴더 선택은 IPC를 통해 메인 프로세스에서 수행한다.
- `필수`: 프리로드에는 범용 브리지가 아니라 목적별 API만 노출한다.
- `권장`: 메인 프로세스에서 입력값, 파일 경로, 권한 범위를 다시 검증한다.

### 3.3 실패를 기본 시나리오로 본다

- `필수`: 긴 작업은 실패, 취소, 중단, 재시도를 고려해 설계한다.
- `권장`: 파일 잠금, 경로 충돌, 권한 거부, 디스크 부족, 포맷 손상 같은 오류를 초기에 모델링한다.
- `권장`: 사용자용 오류 메시지와 개발자용 로그를 분리한다.

## 4. 권장 아키텍처

폴더 구조 예시:

```text
src/
  main/
    app/
    windows/
    ipc/
    services/
    jobs/
    store/
  preload/
    index.ts
  renderer/
    src/
      app/
      pages/
      features/
      components/
      stores/
  shared/
    types/
    contracts/
    constants/
```

책임 분리 기준:

- `main/services`: 파일 시스템, PDF 처리, export, 설정 저장, 업데이트
- `main/jobs`: 장시간 작업 큐, 취소, 진행률, 상태 관리
- `main/ipc`: 채널 등록, 요청 검증, 응답 정규화
- `shared/contracts`: IPC 요청/응답 타입, 이벤트 이름, 에러 코드
- `renderer/features`: 업로드, 미리보기, 내보내기, 설정 등 도메인 단위 UI

구조 규칙:

- `필수`: IPC 이름은 도메인 기준 네임스페이스를 둔다. 예: `dialog:openPdf`, `settings:save`, `job:cancel`
- `필수`: IPC 응답은 직렬화 가능한 평범한 객체로 통일한다.
- `권장`: 비즈니스 로직은 가능한 한 IPC 핸들러 밖의 서비스 계층으로 분리한다.
- `권장`: 장시간 작업은 "작업 시작"과 "진행 상태 통지"를 분리한다.

## 5. BrowserWindow / WebPreferences 기준

기본 예시:

```ts
new BrowserWindow({
  width: 1440,
  height: 960,
  minWidth: 1100,
  minHeight: 700,
  show: false,
  autoHideMenuBar: process.platform === 'win32',
  webPreferences: {
    preload: PRELOAD_PATH,
    contextIsolation: true,
    sandbox: true,
    nodeIntegration: false,
    webSecurity: true
  }
})
```

규칙:

- `필수`: `contextIsolation: true`
- `필수`: `sandbox: true`
- `필수`: `nodeIntegration: false`
- `필수`: `webSecurity: true`를 기본값으로 유지한다.
- `권장`: `show: false`로 생성하고 `ready-to-show` 이후 표시한다.
- `권장`: 개발/배포 환경의 로딩 경로를 명확히 분리한다.
- `권장`: 창 크기, 위치, 최대화 여부 저장 복원을 고려한다.

예외:

- `sandbox: true`는 대부분 권장되지만 일부 네이티브 모듈, 오래된 코드, 특수 브리지 설계와 충돌할 수 있다.
- 그런 경우에도 먼저 구조를 조정해 해결 가능한지 검토하고, 불가피할 때만 예외 사유를 문서화한다.

## 6. 보안 기준

Electron 프로젝트에서 가장 자주 망가지는 부분은 보안과 IPC 경계다.

### 6.1 필수 보안 기준

- `필수`: 렌더러에 Electron/Node API를 직접 노출하지 않는다.
- `필수`: 프리로드에서 최소 API만 `contextBridge`로 공개한다.
- `필수`: IPC 입력값을 검증한다.
- `필수`: IPC 요청의 출처와 사용 가능한 채널을 통제한다.
- `필수`: `webSecurity`를 끄지 않는다.
- `필수`: 외부 링크는 검증 없이 열지 않는다.
- `필수`: Electron 버전을 최신 안정 버전에 가깝게 유지한다.

### 6.2 원격 콘텐츠를 쓰는 경우 추가 기준

- `필수`: HTTPS만 사용한다.
- `필수`: CSP를 둔다.
- `필수`: `ses.setPermissionRequestHandler()` 정책을 둔다.
- `필수`: 탐색 가능한 URL과 새 창 생성 정책을 제한한다.
- `권장`: 원격 콘텐츠와 신뢰 가능한 내부 UI를 세션/권한 단위로 분리한다.

### 6.3 하지 말아야 할 패턴

- `필수`: `contextBridge.exposeInMainWorld('api', { send: ipcRenderer.send })` 형태 금지
- `필수`: 렌더러에 임의 경로 읽기/쓰기 권한 직접 노출 금지
- `필수`: 신뢰되지 않은 URL을 그대로 `shell.openExternal()` 하는 것 금지
- `필수`: 개발용 느슨한 옵션을 운영 빌드에 그대로 가져가는 것 금지

### 6.4 `file://`에 대한 해석

- `권장`: Electron 공식 보안 문서는 `file://` 남용을 피하고 필요 시 custom protocol을 검토하라고 권장한다.
- `선택`: 단순 로컬 UI 앱에서는 `file://` 또는 번들 리소스 로딩을 그대로 써도 된다.
- `필수`: 다만 사용자 입력이나 외부 파일 경로를 그대로 렌더링 경로로 연결하는 구조는 피한다.

즉, `file://` 자체가 무조건 금지인 것은 아니다. 위험한 것은 로딩 경로와 권한 경계를 허술하게 섞는 구조다.

## 7. Preload / IPC 설계 기준

### 7.1 권장 패턴

- 요청/응답: `ipcRenderer.invoke` / `ipcMain.handle`
- 상태 이벤트: `webContents.send`
- 장시간 작업: `job:start`, `job:progress`, `job:completed`, `job:failed`, `job:cancel`

예시 계약:

```ts
type AppErrorCode =
  | 'FILE_NOT_FOUND'
  | 'INVALID_INPUT'
  | 'INVALID_PDF'
  | 'EXPORT_FAILED'
  | 'PERMISSION_DENIED'

interface AppResult<T> {
  ok: boolean
  data?: T
  error?: {
    code: AppErrorCode
    message: string
    details?: unknown
  }
}
```

### 7.2 규칙

- `필수`: 채널 이름은 상수화한다.
- `필수`: 입력 스키마를 검증한다.
- `필수`: 응답 타입을 문서화한다.
- `필수`: OS 원문 에러 메시지를 UI 문자열로 그대로 노출하지 않는다.
- `권장`: 에러 코드는 UI, 로그, 테스트에서 재사용 가능한 단위로 설계한다.
- `권장`: 취소 가능한 작업은 `AbortController` 또는 job 토큰을 둔다.

### 7.3 프리로드 API 규칙

- `필수`: 렌더러에는 "업무 목적별 메서드"만 노출한다.
- `필수`: `openPdf()`, `saveFile()`, `startExport()`처럼 명시적 이름을 쓴다.
- `권장`: 파일 시스템 전체 권한처럼 과도하게 넓은 API는 만들지 않는다.

## 8. 로컬 파일 / PDF 업로드 앱 기준

이 섹션은 이 문서의 핵심 적용 대상이다.

### 8.1 파일 열기

- `필수`: 파일 선택은 메인 프로세스에서 네이티브 다이얼로그로 처리한다.
- `필수`: 확장자 필터만 믿지 말고 실제 파싱 가능 여부를 확인한다.
- `권장`: drag and drop과 파일 선택 버튼을 함께 제공한다.
- `권장`: 최근 파일 목록은 경로 유효성 검사와 함께 제공한다.

PDF 앱 추가 체크:

- `필수`: 손상된 파일, 암호화된 파일, 파싱 실패를 처리한다.
- `권장`: 페이지 수, 페이지 크기, 렌더 스케일에 따른 메모리 상한을 계산한다.
- `권장`: 폰트 누락, 렌더 불가, 비정상 구조 PDF의 실패 케이스를 별도 에러 코드로 분리한다.

### 8.2 저장 / 내보내기

- `필수`: 원본 파일을 직접 수정하지 않는다.
- `필수`: 별도 작업 디렉토리 또는 출력 디렉토리를 사용한다.
- `필수`: 덮어쓰기 정책을 정한다.
- `필수`: 금지 파일명, 예약어, 잘못된 문자 치환 규칙을 둔다.
- `권장`: 사용자 표시명과 실제 저장 파일명을 분리 가능하게 설계한다.

Windows 추가 체크:

- `필수`: `CON`, `PRN`, `AUX`, `NUL`, `COM1` 등 예약어를 피한다.
- `권장`: 너무 긴 경로와 특수문자 조합을 테스트한다.

### 8.3 긴 작업 / 진행률 / 취소

- `필수`: PDF 렌더링, 이미지 변환, 압축, export는 job 단위로 관리한다.
- `필수`: 진행률과 현재 단계를 UI에 노출한다.
- `권장`: 작업 취소를 지원한다.
- `권장`: 앱 종료 중 진행 중 작업 처리 정책을 정한다.

### 8.4 성능 / 메모리

- `필수`: 전체 파일을 한 번에 메모리에 올리는 구조를 피한다.
- `권장`: 페이지 단위 또는 배치 단위로 처리한다.
- `권장`: 미리보기용 저해상도 파이프라인과 최종 export용 고해상도 파이프라인을 분리한다.
- `권장`: 캐시 수명주기를 작업 단위로 명확히 관리한다.

주의:

- CPU 집약 작업을 무조건 메인 프로세스에 몰아넣으면 UI 응답성이 무너질 수 있다.
- 프로젝트 규모에 따라 Worker Threads, child process, 별도 작업 큐를 검토한다.

## 9. 경로 / 저장소 / 데이터 정책

### 9.1 경로 처리

- `필수`: 경로 결합은 항상 Node `path` API를 사용한다.
- `필수`: 구분자 문자열 치환으로 OS 차이를 처리하지 않는다.
- `권장`: 사용자 데이터는 `app.getPath('userData')` 하위에 저장한다.
- `권장`: 임시 파일은 `temp` 또는 앱 전용 임시 디렉토리를 사용한다.

### 9.2 저장 데이터 분리

- `필수`: 사용자 설정, 최근 파일, 창 상태, 캐시, 작업 이력을 분리한다.
- `권장`: 설정 파일 버전과 마이그레이션 경로를 둔다.
- `권장`: 민감값은 평문 저장을 피하고 필요하면 OS 보안 저장소를 검토한다.

### 9.3 로그 정책

- `필수`: 로그 경로와 로그 수준을 정한다.
- `권장`: 사용자용 "로그 폴더 열기" 또는 "로그 내보내기" 기능을 제공한다.
- `권장`: 개인정보, 원본 파일 내용, 민감 경로를 과도하게 로그에 남기지 않는다.

## 10. macOS / Windows 차이

### 10.1 생명주기

- `필수`: `app.whenReady()` 이후 창을 생성한다.
- `필수`: `window-all-closed` 정책을 명확히 둔다.
- `필수`: macOS에서는 `activate`에서 창 재생성 또는 재표시를 처리한다.
- `선택`: 단일 인스턴스 정책은 앱 특성에 따라 적용한다.

실무 해석:

- macOS는 마지막 창을 닫아도 앱이 살아있는 UX가 더 자연스럽다.
- Windows는 마지막 창을 닫으면 앱이 종료되는 UX가 일반적이다.

### 10.2 메뉴 / 단축키

- `필수`: macOS 앱 메뉴 구조는 OS 관례를 따른다.
- `권장`: Windows 메뉴바 표시 전략을 정한다.
- `권장`: `Cmd`와 `Ctrl` 차이를 고려해 단축키를 설계한다.

### 10.3 파일 다이얼로그

- `필수`: 다이얼로그는 메인 프로세스에서 연다.
- `필수`: 가능한 한 부모 창에 연결한다.
- `권장`: macOS는 비동기 다이얼로그 흐름을 전제로 UI를 설계한다.
- `권장`: OS별 파일 선택/폴더 선택 제한 차이를 감안해 UI를 분리할 수 있게 한다.

### 10.4 배포 현실

- `필수`: macOS는 코드 서명과 노타리제이션을 기본 전제로 본다.
- `권장`: Windows는 SmartScreen과 서명 신뢰도 문제를 배포 UX에 포함해 본다.
- `권장`: 설치 경로 권한, UAC, 보안 제품 간섭을 테스트한다.

---

## 11. 배포 / 서명 / 업데이트 기준

### 11.1 패키징 형식

- `권장`: macOS는 `.dmg` 또는 `.zip`
- `권장`: Windows는 `NSIS`, `MSIX`, 필요 시 `portable`

선택 기준:

- 자동 업데이트 필요 여부
- 기업 배포인지 스토어 배포인지
- 설치형/무설치형 요구사항
- 인증서 운영 방식

### 11.2 코드 서명

- `필수`: macOS 배포 앱은 서명 후 노타리제이션 절차를 포함한다.
- `권장`: Windows 배포 앱은 서명 전략을 초기에 결정한다.
- `권장`: Windows는 EV 코드 서명 인증서 또는 Azure Trusted Signing 같은 실무 대안을 검토한다.

주의:

- Windows 서명은 "항상 EV만 정답"은 아니다.
- 조직의 배포 채널, 예산, 신뢰도 요구사항에 맞춰 선택해야 한다.

### 11.3 자동 업데이트

- `권장`: 업데이트는 패키징 방식과 배포 채널을 먼저 정한 뒤 설계한다.
- `권장`: 다운로드, 설치, 재시작 UX를 분리한다.
- `권장`: 네트워크 실패, 롤백, 서명 누락, 피드 오류를 고려한다.

주의:

- Electron 내장 `autoUpdater`는 macOS와 Windows 중심이다.
- 실제 업데이트 전략은 `electron-builder`, `electron-updater`, Squirrel, MSIX, 자체 피드 등 선택에 따라 크게 달라진다.
- 따라서 "autoUpdater를 무조건 넣는다"가 아니라 배포 체인에 맞는 업데이트 방식을 먼저 결정해야 한다.

## 12. 테스트 전략

### 12.1 최소 테스트 기준

- `필수`: 단위 테스트로 경로 유틸, 파일명 규칙, 입력 검증, 핵심 변환 로직을 검증한다.
- `필수`: 통합 테스트로 IPC 핸들러, 설정 저장/복원, 작업 큐, 파일 입출력 경계를 검증한다.
- `필수`: E2E 테스트로 앱 실행, 파일 업로드, 작업 실행, 결과 저장 흐름을 검증한다.

원칙:

- `필수`: 테스트는 구현 이후에 덧붙이는 부가 작업이 아니라 설계 단계부터 함께 정의한다.
- `필수`: 성공 케이스만 아니라 실패 케이스와 경계 조건을 함께 정의한다.
- `권장`: 비즈니스 로직은 unit 테스트가 쉬운 구조로 분리한다.
- `권장`: OS 차이, 경로 문제, 권한 문제처럼 실제 운영에서 자주 깨지는 부분을 통합 테스트와 E2E에 반영한다.

권장 분리:

- `unit`: 순수 함수, 변환 로직, 파일명 정규화, 입력 검증, 에러 코드 매핑
- `integration`: IPC 계약, 설정 저장/복원, 작업 큐, 파일 시스템 경계, 취소 흐름
- `E2E`: 앱 실행, 파일 열기, 작업 수행, 진행률 확인, 저장, 오류 표시

### 12.2 플랫폼 테스트

- `권장`: macOS 최신 안정 버전에서 테스트한다.
- `권장`: Windows 11 기준으로 테스트한다.
- `필수`: 최소 1회는 실제 패키징된 앱 기준 smoke test를 한다.

### 12.3 파일 기반 앱 추가 테스트

- 큰 PDF
- 손상된 PDF
- 암호화된 PDF
- 긴 파일명
- 한글 및 특수문자 파일명
- 읽기 전용 디렉토리
- 외장 디스크 또는 네트워크 드라이브
- 진행 중 작업 취소
- 파일명 충돌과 덮어쓰기 정책
- 저장 경로 없음 또는 권한 거부
- 메모리 사용량이 급증할 수 있는 큰 입력

### 12.4 패키징된 앱 기준 확인

- `필수`: 개발 모드만 아니라 실제 패키징된 앱으로 핵심 흐름을 검증한다.
- `권장`: 설치, 최초 실행, 로그 경로, 저장 경로, 외부 폴더 열기, 업데이트 알림 같은 운영 흐름도 확인한다.
- `권장`: macOS와 Windows 각각에서 최소 1회 이상 smoke test를 수행한다.

## 14. AI 모델에게 줄 구현 지침

아래 텍스트는 AI에게 그대로 전달해도 되는 베이스 프롬프트다.

```text
이 프로젝트는 Electron 기반 데스크톱 앱이다.
macOS와 Windows를 모두 지원해야 한다.

다음 규칙을 반드시 지켜라.

[필수]
1. main / preload / renderer 역할을 분리할 것
2. BrowserWindow 기본값은 contextIsolation=true, nodeIntegration=false, webSecurity=true를 우선 사용할 것
3. sandbox=true를 기본값으로 검토하고, 예외가 필요하면 이유를 명시할 것
4. 파일 열기/저장/폴더 선택은 main 프로세스에서 처리할 것
5. preload에서는 목적별 최소 API만 contextBridge로 노출할 것
6. IPC 채널은 타입과 입력 검증을 포함해 설계할 것
7. 장시간 작업은 job 구조와 진행률 이벤트를 포함할 것
8. 사용자 데이터 저장은 app.getPath('userData') 기준 정책을 설계할 것
9. 보안 편의 옵션(webSecurity off, nodeIntegration on, 범용 IPC 브리지)은 사용하지 말 것
10. macOS와 Windows의 생명주기, 다이얼로그, 배포 차이를 반영할 것
11. 테스트를 부수 결과물이 아니라 핵심 결과물로 취급할 것
12. unit / integration / E2E 테스트 전략 또는 실제 테스트 코드를 함께 제시할 것

[권장]
1. 창 상태 저장/복원
2. 로그 구조와 에러 코드 체계
3. 최근 파일 / 최근 작업
4. 실제 패키징 앱 기준 테스트 전략
5. 코드 서명, macOS 노타리제이션, Windows 서명 전략 제안
6. 실패 케이스와 경계 조건을 포함한 테스트 케이스 제안

[상황별 예외]
1. 원격 콘텐츠를 로드하는 앱이면 CSP, 권한 요청 핸들러, 탐색 제한 정책을 추가할 것
2. 스토어 배포 앱이면 스토어 제약과 샌드박스 정책을 우선 고려할 것
3. sandbox=true가 기술적으로 곤란하면 해제 이유와 보완책을 명시할 것

결과에는 아래를 포함하라.
- 폴더 구조
- IPC 계약
- BrowserWindow 생성 코드
- preload 브리지 예시
- 파일 업로드 및 저장 플로우
- 에러 처리 구조
- 작업 진행률 구조
- 테스트 전략
- 최소 단위 테스트 대상 목록
- 최소 통합 테스트 대상 목록
- 최소 E2E 시나리오 목록

금지사항:
- ipcRenderer 전체를 렌더러에 그대로 노출하지 말 것
- 렌더러에 파일 시스템 전체 권한을 직접 주지 말 것
- 신뢰되지 않은 외부 링크를 검증 없이 열지 말 것
- 배포 보안 기준을 개발 편의 설정으로 대체하지 말 것
- 테스트 없이 핵심 로직만 제안하고 끝내지 말 것
```

## 15. 빠른 체크리스트

### 15.1 프로젝트 시작 전

- 지원 OS와 배포 형식을 결정했는가
- 로컬 파일 앱인지 원격 콘텐츠 앱인지 구분했는가
- 메인/프리로드/렌더러 경계를 정했는가
- 보안 기본값을 고정했는가
- 저장 경로, 캐시, 로그 정책을 정했는가
- 긴 작업 처리 방식을 정했는가

### 15.2 구현 중

- IPC 입력 검증을 하고 있는가
- 렌더러에 과도한 권한을 주지 않았는가
- 큰 파일 메모리 사용량을 통제하고 있는가
- macOS/Windows 차이를 UX에 반영했는가
- 에러 코드와 사용자 메시지를 분리했는가

### 15.3 배포 전

- 코드 서명과 인증서 운영이 준비됐는가
- macOS 노타리제이션 절차가 포함됐는가
- Windows 설치/업데이트 동작을 확인했는가
- 패키징된 앱 기준 smoke test를 했는가
- 로그 수집과 장애 대응 경로를 준비했는가
